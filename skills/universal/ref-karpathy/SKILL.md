# Karpathy 코딩 가이드라인

## 1. 코딩 전에 생각하라

**가정하지 말 것. 혼란을 숨기지 말 것. 트레이드오프를 드러낼 것.**

구현 전 반드시:
- 가정을 명시적으로 기술한다. 불확실하면 질문한다.
- 해석이 여러 개 가능하면 모두 제시한다 — 임의로 하나를 고르지 않는다.
- 더 단순한 접근법이 있으면 말한다. 필요하면 반론을 제기한다.
- 무언가 불명확하면 멈춘다. 무엇이 혼란스러운지 명시하고 질문한다.

## 2. 단순함 우선

**문제를 해결하는 최소한의 코드. 추측성 코드 금지.**

- 요청된 것 이상의 기능을 추가하지 않는다.
- 한 번만 쓰이는 코드에 추상화를 만들지 않는다.
- 요청되지 않은 "유연성"이나 "설정 가능성"을 넣지 않는다.
- 발생 불가능한 시나리오에 대한 에러 처리를 넣지 않는다.
- 200줄로 작성했는데 50줄로 가능하면, 다시 작성한다.

자문: "시니어 엔지니어가 이걸 보고 과잉 설계라고 할까?" 그렇다면 단순화한다.

## 3. 외과적 변경

**필요한 것만 수정한다. 자기가 만든 잔해만 정리한다.**

기존 코드를 수정할 때:
- 인접 코드, 주석, 포맷팅을 "개선"하지 않는다.
- 고장나지 않은 것을 리팩토링하지 않는다.
- 자신이 다르게 할 스타일이라도 기존 스타일에 맞춘다.
- 관련 없는 dead code를 발견하면 언급은 하되 삭제하지 않는다.

내 변경으로 고아가 발생했을 때:
- 내 변경이 미사용으로 만든 import/변수/함수는 제거한다.
- 기존에 이미 존재하던 dead code는 요청 없이 제거하지 않는다.

**검증 기준: 변경된 모든 줄이 사용자의 요청에 직접 추적 가능해야 한다.**

## 4. 목표 지향 실행

**성공 기준을 정의하고, 검증될 때까지 반복한다.**

작업을 검증 가능한 목표로 변환:
- "유효성 검사 추가" → "잘못된 입력에 대한 테스트를 작성하고 통과시킨다"
- "버그 수정" → "재현 테스트를 작성하고 통과시킨다"
- "X 리팩토링" → "리팩토링 전후로 테스트가 통과하는지 확인한다"

다단계 작업 시 간결한 계획을 명시:
```
1. [단계] → 검증: [확인 방법]
2. [단계] → 검증: [확인 방법]
3. [단계] → 검증: [확인 방법]
```

강한 성공 기준은 자율적 반복을 가능하게 한다. 약한 기준("되게 해줘")은 끊임없는 확인이 필요하다.
